# Return Guardian 추가 기능 후보 검토

## 우선순위 A: MVP 직후 바로 가치가 큰 기능

1. JSON import
   - 이유: 로컬 앱은 브라우저 저장소가 지워질 수 있으므로 export만으로는 부족하다.
   - 범위: MVP JSON export를 다시 불러와 purchases, receipts, attachments metadata, reminders를 복원한다.
   - 주의: MVP JSON은 첨부 파일 본문을 포함하지 않으므로 import 후 첨부 파일은 "파일 재첨부 필요" 상태로 표시한다.

2. 전체 백업 zip export
   - 이유: 영수증 이미지/PDF까지 함께 보관해야 진짜 백업이 된다.
   - 범위: JSON 데이터 + IndexedDB Blob 파일을 zip으로 묶는다.
   - 주의: 브라우저 메모리 사용량과 대용량 파일 처리가 핵심 리스크다.

3. `.ics` 캘린더 export
   - 이유: 사용자는 반품/보증 마감일을 기존 캘린더에서 보고 싶어 한다.
   - 범위: 구매별 또는 전체 마감일을 `.ics`로 내보낸다.
   - 주의: 자동 캘린더 동기화가 아니라 파일 export로 시작한다.

4. 브라우저 알림
   - 이유: 대시보드에 들어오지 않아도 오늘 마감 항목을 알 수 있다.
   - 범위: PWA 또는 브라우저 notification permission 기반 알림.
   - 주의: 로컬 앱에서는 사용자가 브라우저를 열지 않으면 알림 신뢰성이 제한될 수 있다.

## 우선순위 B: 제품 완성도를 높이는 기능

1. OCR 영수증 추출
   - 날짜, 구매처, 금액 후보를 추출하되 사용자가 확인 후 저장한다.
   - MVP 철학상 자동 저장보다 "제안 후 확인"이 안전하다.

2. 로컬 암호화 저장
   - 구매 정보와 영수증은 민감할 수 있다.
   - 사용자가 암호를 잊으면 복구할 수 없으므로 UX 문구와 백업 흐름이 중요하다.

3. 제품 매뉴얼/시리얼/수리 이력
   - 보증 청구 시 실제로 필요한 정보까지 연결한다.
   - purchase detail 화면에 별도 섹션으로 확장하는 것이 자연스럽다.

4. 마감 기준 커스터마이즈
   - "보증 곧 만료" 기준 30일을 7/14/30/60/90일로 변경한다.
   - settings에 저장하면 구현 난이도는 낮다.

## 우선순위 C: 장기 후보

1. 이메일/쇼핑몰 영수증 가져오기
   - 가치가 크지만 개인정보, 인증, API 유지보수 부담이 크다.

2. 클라우드 동기화
   - 여러 기기 사용성은 좋아지지만 "서버 업로드 없음"이라는 핵심 신뢰 메시지를 바꾼다.

3. 모바일 앱 배포
   - 초기 범위 밖이다. PWA 사용성이 충분히 검증된 뒤 판단한다.

4. 가격 추적/중고가 비교
   - 구매 후 반품 판단에 도움이 될 수 있지만 제품 정체성이 넓어질 위험이 있다.

## 추천 다음 개발 순서

1. JSON import
2. `.ics` export
3. 전체 백업 zip export
4. PWA install + 브라우저 알림
5. OCR 후보 추출
6. 로컬 암호화 저장

가장 먼저 추가할 기능은 JSON import가 맞다. 로컬 우선 앱에서 export는 이미 있으므로, 복원 기능이 있어야 사용자가 안심하고 실제 구매 데이터를 넣기 시작할 수 있다.
